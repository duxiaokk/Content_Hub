# Content Hub MVP - 仓库级 Agent 规则

## 1. 适用范围
- 本文件适用于整个仓库 `D:\Python\content_hub`。
- 若子目录存在更具体的 `AGENTS.md`，子目录规则可补充细节，但不得与本文件核心流程冲突。

## 2. 每次任务开始前必须读取
1. 读取根目录 `codex_board.md`
2. 若看板存在 `IN PROGRESS` 任务，读取对应 `tasks/Txxx*.md`
3. 若用户显式指定任务，仍需先读取看板，再读取对应任务文件
4. 未读取上述文件前，不得直接开始写代码

## 3. 单任务闭环流程

### Step 1: 侦查
- 阅读任务文件中列出的输入文件
- 输出不超过 5 行的现状摘要
- 标出会受影响的模块、接口、模型或测试

### Step 2: 方案
- 明确列出：
  - 修改文件
  - 新增文件
  - 关键类 / 函数 / 接口
  - 验证命令
- 进入实现前必须等待用户回复“确认”
- `确认`、`ok`、`直接做` 视为确认

### Step 3: 实现
- 保持与仓库现有风格一致
- 仅实现当前任务要求，不扩散重构
- 不引入任务未要求的新依赖
- 优先复用现有运行时、模型、注册表、工具函数

### Step 4: 验证
- 必须执行任务文件中的验收命令
- 验证失败时先修复，再重新验证
- 输出验证结果摘要，包括通过项和遗留问题

### Step 5: 收尾
- 更新 `codex_board.md`
  - 当前任务标记为 `DONE`
  - 下一个任务标记为 `IN PROGRESS`
- 给出变更摘要：
  - 修改文件
  - 新增文件
  - 已知问题
  - commit 建议

## 4. Git 提交规则
- 每个任务在“验证通过且看板已更新”后，自动执行 `git add` 和 `git commit`
- commit message 必须使用格式：`Txxx: 简短描述`
- 例如：`T005: add reddit fetcher`
- 只提交当前任务相关文件
- 不得把无关变更、用户未要求的临时文件、其他任务残留一并提交
- 若工作区存在无关脏文件，提交前应显式避开

## 5. 代码与架构约束

### Python
- 优先使用类型注解
- 异步链路优先使用 `async` / `await`
- 异常需要被捕获并转成可定位的问题信息
- 日志和错误消息使用英文，注释使用中文

### 数据层
- 使用 SQLAlchemy 模型与 Alembic 迁移
- 字段命名、默认值、状态枚举遵循现有模型风格
- 迁移链必须保持单 head，避免分叉

### API 与服务
- FastAPI 风格
- 输入输出结构尽量使用现有 schema / contract
- fetch / process / publish 各层统一走现有注册机制，不重复造抽象

### 测试
- 优先在现有测试文件中补充当前任务覆盖
- 测试覆盖规模与改动风险相匹配
- 对外部网络依赖优先用 stub / mock 保持稳定

## 6. MVP 阶段红线
- 不引入 Redis、RabbitMQ、Kafka 等基础设施，除非任务明确要求
- 不增加多租户、复杂权限、OAuth2
- 不重写现有目录结构
- 不为“未来扩展”提前引入插件系统或抽象基类
- 不把单任务扩散成跨模块重构

## 7. 项目特定注意事项
- 当前任务推进以 `codex_board.md` 为准
- 现阶段主链路是：
  - `fetcher_engine`
  - `ai_processor`
  - `workflow_engine`
  - `platform`
  - `publisher_engine`
- `FetchService` 相关抓取器统一返回 `list[SourceItem]`
- 若默认 SQLite 工作库出现 `disk I/O error`，应优先说明环境问题，并使用干净临时库完成迁移/验收验证
- `pytest-cache-files-*` 权限 warning 目前可能持续出现，若测试通过，可作为已知问题记录

## 8. 输出要求
- 方案阶段回复必须包含 `## 下一步`
- 实现阶段回复必须包含 `## 当前进度`
- 收尾阶段回复必须包含 `## 变更摘要`

## 9. 语言要求
- 与用户交流使用中文
- 代码注释使用中文
- 日志、异常、测试断言中的错误说明优先使用英文
