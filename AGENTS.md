# Content Hub MVP - 仓库级 Agent 规则

## 1. 适用范围
- 本文件适用于整个仓库 `D:\Python\content_hub`。

## 2. 每次任务开始前必须读取
1. 读取根目录 `change.md`，了解近期变更上下文
2. 若存在待办任务文件（如 `tasks/Txxx*.md`），读取对应任务文件
3. 未读取上述文件前，不得直接开始写代码

## 3. 单任务闭环流程

### Step 1: 侦查
- 阅读任务文件中列出的输入文件
- 输出不超过 5 行的现状摘要
- 标出会受影响的模块、接口、模型或测试

### Step 2: 方案
- 明确列出：修改文件、新增文件、关键接口、验证命令
- 进入实现前必须等待用户回复 `确认` / `ok` / `直接做`

### Step 3: 实现
- 保持与仓库现有风格一致
- 仅实现当前任务要求，不扩散重构
- 不引入任务未要求的新依赖
- 优先复用现有运行时、模型、注册表、工具函数

### Step 4: 验证
- 执行验收命令，验证失败先修复再重试
- 输出验证结果摘要，包括通过项和遗留问题

### Step 5: 收尾
- 更新 `change.md`，记录本次改动的文件及原因（追加到文件顶部）
- 给出变更摘要：修改文件、新增文件、已知问题、commit 建议
- 自动执行 `git add` 和 `git commit`

## 4. Git 提交规则
- commit message 格式：`Txxx: 简短描述`
- 只提交当前任务相关文件，无关脏文件显式避开

## 5. 代码与架构约束

### Python
- 优先使用类型注解，异步链路优先使用 `async` / `await`
- 日志和错误消息使用英文，注释使用中文
- 异常被捕获并转成可定位的问题信息

### 数据层
- SQLAlchemy 模型 + Alembic 迁移，迁移链保持单 head

### API 与服务
- FastAPI 风格，输入输出使用现有 schema / contract
- fetch / process / publish 各层统一走现有注册机制

### 测试
- 优先在现有测试文件中补充当前任务覆盖
- 外部网络依赖优先用 stub / mock

## 6. MVP 阶段红线
- 不引入 Redis / RabbitMQ / Kafka 等基础设施（除非任务明确要求）
- 不增加多租户、复杂权限、OAuth2
- 不引入任务未要求的插件系统或抽象基类
- 不把单任务扩散成跨模块重构

## 7. 语言要求
- 与用户交流使用中文
- 代码注释使用中文
- 日志、异常、测试断言中的错误说明优先使用英文

## 8. 变更记录规则
- 每次代码改动后，必须在根目录 `change.md` 中追加记录，格式如下：
  - 日期、改动文件、改动原因（一句话）
- `change.md` 按时间倒序排列，最新记录写在文件顶部
- 禁止遗漏：即使只改了一个文件，也必须记录
