# Content Hub MVP - 仓库级 Agent 规则

## 1. 适用范围
- 本文件适用于整个仓库 `D:\Python\content_hub`。
- 目标是让每次改动都保持单任务闭环、边界清晰、兼容层可收敛、验证可复现。

## 2. 每次任务开始前必须读取
1. 读取根目录 `change.md`，了解近期变更上下文。
2. 若存在待办任务文件（如 `tasks/Txxx*.md`），读取对应任务文件。
3. 若任务涉及某个模块，补读该模块的入口、README、测试或设计说明。
4. 未完成上述阅读前，不得直接开始写代码。

## 3. 单任务闭环流程

### Step 1: 侦查
- 阅读任务文件和相关输入文件。
- 输出不超过 5 行的现状摘要。
- 标出会受影响的模块、接口、模型、脚本或测试。
- 如发现重复目录、双份实现、历史兼容层或命名混乱，必须先显式记录，不得跳过。

### Step 2: 方案
- 明确列出：修改文件、新增文件、关键接口、验证命令、兼容影响。
- 若改动会触及旧目录、兼容入口、迁移脚本或构建链路，必须说明保留原因或收敛路径。
- 进入实现前必须等待用户回复 `确认` / `ok` / `直接做`。

### Step 3: 实现
- 保持与仓库现有风格一致。
- 仅实现当前任务要求，不扩散重构。
- 不引入任务未要求的新依赖。
- 优先复用现有运行时、模型、注册表、工具函数。
- 若发现需要重构的历史问题超出本任务范围，记录为后续债务，不在本任务顺手扩散。

### Step 4: 验证
- 执行验收命令，验证失败先修复再重试。
- 输出验证结果摘要，包括通过项、失败项和遗留问题。
- 构建、测试、缓存、权限、编码等环境噪音不能直接忽略，至少要说明现象、范围和是否阻塞交付。

### Step 5: 收尾
- 更新 `change.md`，记录本次改动的文件及原因，追加到文件顶部。
- 给出变更摘要：修改文件、新增文件、验证结果、已知问题、后续建议。
- 仅在用户明确要求提交代码时，才执行 `git add` 和 `git commit`。

## 4. Git 提交规则
- commit message 格式：`Txxx: 简短描述`。
- 只提交当前任务相关文件，无关脏文件显式避开。
- 未经用户明确要求，不自动创建 commit、不改写历史、不清理用户未授权的脏改动。

## 5. 架构与目录约束

### 5.1 单一权威实现
- 同一能力只能保留一个权威实现目录；禁止把同一模块以 copy 方式并行放在多个目录长期演进。
- 若历史上已存在重复目录，新增改动必须优先落到权威目录；非权威目录只允许做迁移、转发、只读兼容或删除准备。
- 如 `ado_repost`、`content_bridge`、`shared_mempool`、`shared_memory` 这类命名分叉仍存在，实现前必须先判断本次改动应落在哪个权威位置。

### 5.2 兼容层治理
- 标记为 `legacy`、`frozen`、`compatibility only` 的模块视为冻结层，不得继续堆叠新功能。
- 兼容层只允许做薄转发、兼容告警、迁移兜底和删除前过渡。
- 若必须修改兼容层，提交说明中必须写清：为什么不能直接改主实现、兼容层何时可以移除。

### 5.3 导入与模块边界
- 禁止在常规业务代码中大面积使用 `try/except ImportError` 做包路径回退。
- 确有历史兼容需要时，必须把回退逻辑集中到单独的 bridge/shim 模块，不得在多个业务文件重复散落。
- 新代码必须使用稳定、单一、可预测的包路径；不要同时支持多套 import 入口。

## 6. 代码与风格约束

### Python
- 优先使用类型注解，异步链路优先使用 `async` / `await`。
- 日志和错误消息使用英文，注释使用中文。
- 异常被捕获后要转成可定位的问题信息，避免静默吞错。

### 注释与文档
- 注释应短而必要，只解释非显然约束、兼容原因、边界条件和坑点。
- 禁止在源码文件头保留“大段初学者指南”“入门教程式”注释；这类内容应移到 README 或专门文档。
- 文档要描述当前事实，不要让 README、注释、脚本名称继续为历史目录或旧命名背书。

### 命名与编码
- 代码标识符、目录名、模块名、schema 字段名、环境变量名默认使用英文和 ASCII。
- 中文只用于必要注释、任务说明和面向团队的文档，不用于公共接口字段或跨模块契约。
- Markdown、源码、配置文件统一使用 UTF-8 编码；发现乱码文件时，修复前不得继续扩散内容。

## 7. 数据层与接口约束

### 数据层
- SQLAlchemy 模型 + Alembic 迁移，迁移链保持单 head。
- 不允许为绕过迁移而在运行时偷偷补字段、改表名或做隐式结构兼容。

### API 与服务
- FastAPI 风格，输入输出使用现有 schema / contract。
- fetch / process / publish 各层统一走现有注册机制。
- 规划、编排、聚合等旧层若已标记冻结，新能力应优先接入当前主路径，不再倒灌回旧编排层。

## 8. 测试与环境验证
- 优先在现有测试文件中补充当前任务覆盖。
- 外部网络依赖优先用 stub / mock。
- 涉及前端构建、Python 测试、数据库迁移、脚本执行时，方案中应列出对应验证命令。
- 对 `vite build` 失败、`pytest` cache 权限警告、编码问题、路径问题等环境兼容性噪音，要给出可执行的规避或治理建议，不能只报现象。

## 9. MVP 阶段红线
- 不引入 Redis / RabbitMQ / Kafka 等基础设施，除非任务明确要求。
- 不增加多租户、复杂权限、OAuth2。
- 不引入任务未要求的插件系统或抽象基类。
- 不把单任务扩散成跨模块重构。
- 不借“兼容”之名继续复制目录、复制模块、复制运行时。

## 10. 语言要求
- 与用户交流使用中文。
- 代码注释使用中文。
- 日志、异常、测试断言中的错误说明优先使用英文。

## 11. 变更记录规则
- 每次代码改动后，必须在根目录 `change.md` 中追加记录。
- 记录格式：`日期（精确到秒） | Agent/模型 | 改动文件 | 改动原因`。
- 示例：
  ```
  2025-06-21 18:09:32 | kimi | AGENTS.md, change.md | 补充变更记录规则：日期精确到秒，增加模型标识字段
  ```
- `change.md` 按时间倒序排列，最新记录写在文件顶部。
- 禁止遗漏：即使只改了一个文件，也必须记录。
