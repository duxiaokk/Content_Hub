# Content Hub 架构说明

## 1. 系统组成

Content Hub 目前是一个多应用工作区，由应用层、共享库层和基础设施层组成。

### 1.1 应用层

应用层位于 `apps/`，包含以下 3 个主要服务：

- `platform`
  - 主平台应用
  - 负责平台页面、平台 API、内容管理和部分调度入口

- `comment_agent`
  - 评论服务
  - 负责接收事件、管理评论相关任务，并提供独立接口

- `ado_repost`
  - 当前承载 `content_bridge` 的代码目录
  - `content_bridge` 是外部内容搬运与预处理服务，负责从外部平台抓取博主动态，并进行预处理

### 1.2 共享库层

共享库层位于 `libs/`，当前包含：

- `shared_memory`
  - 当前正式共享库
  - 用于承载多个服务可复用的共享能力

- `shared_mempool`
  - 历史参考目录
  - 当前不应继续作为新开发主入口

### 1.3 基础设施层

基础设施层位于 `infra/`，当前包含：

- `docs/`
  - 工作区级说明文档

- `docker/`
  - 容器和部署相关文件

- `scripts/`
  - 工作区级启动、初始化和辅助脚本

## 2. 模块职责

### 2.1 platform

`platform` 是工作区中的主平台服务，承担统一入口角色。

从当前目录结构看，它主要负责：

- 平台页面与前端资源组织
- 平台业务 API
- 内容与数据模型管理
- 部分调度、任务和服务编排入口

它更像“面向用户或管理端的主应用”。

### 2.2 comment_agent

`comment_agent` 是独立的评论服务。

从当前仓库现状看，它主要负责：

- 接收评论相关事件
- 管理站点、代理、任务等评论相关数据
- 对外暴露基础 API
- 为后续异步执行或自动化评论流程预留入口

它更像“面向评论流程的专用后端服务”。

### 2.3 content_bridge

`content_bridge` 是外部内容搬运与预处理服务，当前代码目录仍为 `apps/ado_repost`。

从当前目录和子文档描述看，它主要负责：

- 从外部平台抓取博主动态或内容更新
- 支持增量抓取
- 对抓取结果做去重、翻译、格式化、摘要或投递前整理

它更像“外部内容进入系统之前的桥接与清洗服务”。

### 2.4 shared_memory

`shared_memory` 是当前统一共享库。

它的职责不是直接对外提供业务页面，而是为多个服务提供公共能力，例如：

- 可复用的数据存储能力
- 共享工具函数或适配逻辑
- 跨服务共用的基础模块

## 3. 调用关系

当前系统更适合理解为“一个主平台 + 两个专用服务 + 一个共享库”。

### 3.1 高层关系

```text
Browser / Client
       |
       v
   platform
    /    \
   v      v
comment  ado_repost
 agent
    \    /
     v  v
 shared_memory
```

这张图表达的是高层调用方向，不代表所有调用都已经完全标准化。

### 3.2 关系说明

1. `platform` 是工作区对外最核心的入口
   - 用户或管理端更可能先接触它
   - 它负责整合页面、业务流程和部分服务入口

2. `comment_agent` 和 `content_bridge` 是相对独立的专用服务
   - 它们围绕特定业务职责运行
   - 不应把所有业务逻辑都堆回 `platform`

3. `shared_memory` 为多个服务提供复用能力
   - 它应该是“共享底座”
   - 不应该再由历史目录承担正式职责

## 4. 数据流理解

从当前项目职责划分来看，可以先用下面的方式理解主要数据流：

### 4.1 评论相关流

```text
外部事件 / 平台操作
        |
        v
  comment_agent
        |
        v
 评论任务 / 状态 / 共享数据
```

### 4.2 内容采集相关流

```text
外部内容源
    |
    v
content_bridge
    |
    v
抓取结果 / 处理结果 / 游标数据
```

### 4.3 平台整合流

```text
用户访问 / 管理操作
        |
        v
     platform
        |
        v
平台数据、任务入口、服务协作
```

这里的“数据流”，你可以先简单理解成“信息大概从哪里来，经过哪个服务，再变成什么结果”。

## 5. 当前架构问题

当前架构已经有基本拆分，但还存在以下明显问题：

1. 服务边界已出现，但文档边界还不清楚
   - 需要依赖后续文档继续补齐

2. 共享库收敛未完成
   - `shared_memory` 与 `shared_mempool` 并存，容易让开发者误判正式入口

3. 命名与目录仍未完全一致
   - 正式服务名称已收敛为 `content_bridge`
   - 当前代码目录仍为 `apps/ado_repost`

4. 运行关系仍偏“迁移期约定”
   - 当前依赖启动脚本和人工理解
   - 后续需要进一步固化为清晰配置和标准运行方式

## 6. 当前结论

目前的 Content Hub 不应被理解成“一个已经完全定型的微服务平台”，而应被理解成：

- 一个已经完成初步分层的统一工作区
- 一个正在从历史目录迁移到正式结构的工程
- 一个以 `platform` 为主入口、以 `comment_agent` 和 `content_bridge` 为专用服务、以 `shared_memory` 为共享底座的系统

后续架构工作的重点，不是马上重写服务，而是继续把职责、命名、共享库和运行方式收敛清楚。
