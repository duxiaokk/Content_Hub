# Content Hub 迁移状态说明

## 1. 文档目的

本文档用于说明 Content Hub 当前哪些目录已经是正式目录，哪些目录仍处于过渡状态，以及后续的收敛方向。

这份说明的作用是减少两个常见问题：

1. 新人不知道应该优先修改哪个目录
2. 历史目录长期保留后，正式入口和参考副本混在一起

## 2. 当前目录状态

### 2.1 正式目录

以下目录应视为当前工作区的正式目录：

- `apps/platform`
- `apps/comment_agent`
- `apps/ado_repost`
- `libs/shared_memory`
- `infra/docs`
- `infra/scripts`
- `infra/docker`

含义：

- 新功能开发应优先在这些目录内继续推进
- 新文档、脚本和共享能力应优先落在这些目录中
- 其中 `apps/ado_repost` 当前承载的正式服务名称已调整为 `content_bridge`

### 2.2 过渡目录

以下目录当前属于过渡或历史参考状态：

- `libs/shared_mempool`

当前定位：

- 可用于对照历史实现
- 不应继续作为新开发的正式依赖入口
- 不应继续扩大使用范围

## 3. 当前收敛结论

从目录治理角度看，当前工作区已经形成了明确方向：

1. `apps/` 是应用主入口
2. `libs/shared_memory` 是共享库主入口
3. `infra/` 是工作区级文档、脚本和部署入口
4. `libs/shared_mempool` 仅保留为历史参考，不再承担正式职责
5. `content_bridge` 当前仍暂时使用 `apps/ado_repost` 作为外层目录，但运行入口已切换到 `content_bridge` 包名

## 4. 对开发者的直接要求

在迁移完全结束前，建议直接遵循以下规则：

1. 新共享能力优先进入 `libs/shared_memory`
2. 不要新增对 `libs/shared_mempool` 的正式引用
3. 修改历史逻辑时，优先判断能否迁回正式目录，而不是继续堆积在历史目录
4. 工作区级说明统一放在 `infra/docs`
5. 工作区级启动脚本统一放在 `infra/scripts`

## 5. 后续收尾目标

后续迁移收尾至少应完成以下事项：

1. 识别 `shared_mempool` 是否仍被任何运行路径引用
2. 将仍有价值的共享逻辑并入 `shared_memory`
3. 删除或归档不再需要的历史副本
4. 在依赖文件和文档中移除对历史目录的暗示

## 6. 文档编码约定

从当前终端读取效果看，仓库内中文文档存在编码显示异常。

后续统一约定如下：

1. 工作区文档统一使用 UTF-8 编码
2. 新增 Markdown 文档统一保存为 UTF-8
3. 后续逐步修复历史文档的编码不一致问题

这项约定虽然不改变业务逻辑，但会直接影响文档可读性和协作效率。
