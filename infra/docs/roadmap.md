# Content Hub 整改路线图

## 1. 当前问题概述

当前仓库已经完成了基础目录拆分，形成了 `apps`、`libs`、`infra` 三层结构，但整体状态仍然更接近“内部迁移中的工作区”，而不是“适合协作和持续迭代的工程项目”。

目前主要问题集中在以下几个方面：

1. 文档不足
   - 根目录 `README` 不能清楚回答“这是什么、能做什么、怎么跑起来”。
   - 缺少架构说明、开发指南、环境变量说明和部署说明。
   - 新成员很难快速理解三个应用和共享库之间的关系。

2. 迁移状态未收敛
   - `libs/shared_memory` 与 `libs/shared_mempool` 并存，正式共享库边界不够明确。
   - README 中保留了较多“历史参考副本”“便于回滚”的描述，说明迁移工作尚未完全收尾。

3. 可移植性不足
   - 文档中直接出现本机绝对路径，例如 `D:\Python\content_hub`。
   - 如果项目换一台电脑、换一个用户目录或部署到服务器，现有说明无法直接复用。

4. 工程化基础薄弱
   - 依赖管理方式不清晰，尚未明确每个子应用如何安装依赖。
   - 没有可见的测试、CI、格式化或静态检查约定。
   - 启动脚本存在，但缺少统一的运行规范和故障排查入口。

5. 命名与职责表达不清
   - `ado_repost` 的业务含义不够直观，且已无法覆盖未来“多平台博主动态搬运”的定位。
   - `comment_agent` 是否为 AI Agent、普通评论服务或评论处理流程，当前文档没有明确说明。

## 2. 整改目标

本次整改不以“大规模重构代码”为第一目标，而是优先建立一个稳定、可理解、可协作的项目基础。

整改完成后，项目应达到以下目标：

1. 任何新成员进入仓库后，可以在 10 分钟内理解项目用途和基础结构。
2. 任何开发者可以根据文档完成本地环境初始化和最小启动。
3. 每个应用和共享库的职责边界明确，不再依赖“历史参考目录”进行理解。
4. 项目配置不依赖作者本机绝对路径，具备基本可移植性。
5. 至少建立最小测试与检查机制，保证后续调整不会完全失控。

## 3. 分阶段执行计划

### 阶段一：文档补齐

目标：先让项目“能被人看懂”。

执行项：

1. 重写根目录 `README.md`
   - 增加项目简介
   - 增加三个应用的职责说明
   - 增加目录结构说明
   - 增加快速启动步骤
   - 增加开发环境要求

2. 在 `infra/docs` 下补充以下文档
   - `architecture.md`：说明系统组成、模块关系、调用方向
   - `development.md`：说明本地开发流程
   - `environment.md`：说明环境变量和配置项

3. 明确说明当前迁移状态
   - 哪些目录是正式目录
   - 哪些目录是过渡目录
   - 过渡目录后续如何下线

验收标准：

1. 根 README 能回答“这是什么、能做什么、怎么跑起来”。
2. 文档中不再以作者本机路径作为主要说明方式。
3. 新人不看源码，也能理解主要目录职责。

### 阶段二：依赖与启动规范化

目标：让项目“能稳定跑起来”。

执行项：

1. 为每个子应用明确依赖声明方式
   - 优先选择 `pyproject.toml`，或补齐 `requirements.txt`
   - 标明 Python 版本要求

2. 梳理启动入口
   - 统一脚本命名
   - 说明每个脚本用途
   - 标明是否存在严格启动顺序，以及原因

3. 补充 `.env.example`
   - 给出必要配置项的示例值
   - 区分必填项和可选项

验收标准：

1. 每个应用都有明确依赖安装方式。
2. 开发者可以根据文档完成环境初始化。
3. 启动失败时，能快速定位是依赖问题、配置问题还是服务顺序问题。

### 阶段三：命名收敛与共享库治理

目标：让项目“边界清楚，不靠猜”。

执行项：

1. 重新评估 `ado_repost` 命名
   - 根据真实业务，将其收敛为更通用的正式名称 `content_bridge`
   - 使服务定位从“单一来源 repost”升级为“外部内容搬运与预处理”

2. 明确 `comment_agent` 的服务性质
   - 是否调用大模型
   - 输入输出是什么
   - 它对外提供什么能力

3. 收敛共享库
   - 确认 `shared_memory` 为正式共享库
   - 评估 `shared_mempool` 是否仍有引用
   - 制定合并、迁移或删除计划

验收标准：

1. 核心目录名称能够表达真实业务含义。
2. 新开发者不需要翻历史目录判断“应该改哪里”。
3. 共享代码只保留一个正式主入口。

### 阶段四：最小工程化补齐

目标：让项目“可以持续维护”。

执行项：

1. 增加基础测试
   - 每个服务至少补一个核心流程测试
   - 为共享库补一个基础导入或工具函数测试

2. 增加代码质量工具
   - 选择并接入格式化工具
   - 选择并接入静态检查工具

3. 增加 CI
   - 安装依赖
   - 运行测试
   - 运行基础检查

验收标准：

1. 关键模块修改后，能通过自动化测试发现明显回归。
2. 团队成员提交代码时，有统一的基本检查标准。
3. 仓库具备最小持续集成能力。

### 阶段五：部署与交付能力补齐

目标：让项目“可以交给别人运行”。

执行项：

1. 为服务补齐容器化入口
   - 每个服务至少有基础 `Dockerfile`

2. 增加本地编排文件
   - 使用 `docker-compose.yml` 串联服务和依赖

3. 补充运维说明
   - 日志位置
   - 常见故障
   - 健康检查方式

4. 补充仓库协作文档
   - `LICENSE`
   - `CONTRIBUTING.md`
   - `CHANGELOG.md`

验收标准：

1. 第三方开发者可以在非作者机器上启动系统。
2. 项目具备基础对外交付条件。
3. 仓库从“个人工作目录”升级为“可协作工程仓库”。

## 4. 推荐执行顺序

建议按以下顺序推进，不建议并行大范围改造：

1. 先修正文档和说明
2. 再补依赖与启动规范
3. 然后处理命名和共享库收敛
4. 最后补测试、CI 和部署能力

原因很简单：如果在文档、依赖、边界都还不清楚时就直接重构代码，后续很容易反复返工。

## 5. 一期整改完成定义

如果当前只做一期，建议把范围控制在以下内容：

1. 完成 README 重写
2. 补齐架构文档、开发文档、环境变量文档
3. 明确正式目录与过渡目录
4. 为三个应用补齐依赖声明
5. 去除文档中的绝对路径依赖

当以上五项完成后，项目就能从“只能作者自己维护”，提升到“团队内部可以接手和继续开发”的状态。
